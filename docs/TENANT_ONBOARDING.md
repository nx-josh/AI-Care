# Tenant 온보딩 & KB 학습 플로우

**문서 상태**: v0.1 초안
**최종 수정일**: 2026-05-19
**관계 문서**: `PRD.md` §8.3 (RAG), `AI_POLICY.md` §6 (다국어), `PII_MASKING.md` §3.4 (RAG 입력 차단), `RBAC_MATRIX.md` §3.4 (KB 권한)
**대상 검토자**: CS 운영팀 / tenant_admin (게임사 측) / 보안팀

본 문서는 **새 게임이 CS Agent 시스템에 처음 입주할 때 어떻게 학습 자료를 주입하고 활성화하는지**의 운영 플로우. 출시 후 패치·콘텐츠 갱신 사이클도 포함.

> "학습"이라는 표현은 LLM 모델 자체 fine-tuning이 아니라 **RAG 인덱스에 게임 콘텐츠 등록 + 시스템 프롬프트 게임별 커스터마이징**을 의미.

---

## 0. 핵심 원칙

1. **KB는 tenant 소유** — CROSS는 인프라(저장·인덱싱·검색)만 제공.
2. **Citation 강제** — KB 인덱싱 시 출처 가능 단위로 분해. 단일 거대 문서 인덱싱 금지.
3. **자동 PII 차단** — 작성자가 실수해도 시스템이 인덱스 등록 거부.
4. **승인자 ≠ 작성자 (SoD)** — RBAC §0.6 직무 분리.
5. **빈 KB로 출시 금지** — Sandbox 테스트 임계 미달이면 production 활성화 차단.
6. **만료 콘텐츠 자동 정리** — 시즌·이벤트 종료 시 자동 비활성.
7. **buyer-side citation** — 답변에 출처 부착 가능한 형태로만 KB 작성.

## 1. 전체 플로우 개요

```
[A. 초기 셋업 — 출시 전 1회]
   1. Tenant 생성 (계약·DPO 통과)
        │
   2. 메타데이터 등록 (게임·언어·결제·톤)
        │
   3. 콘텐츠 수집 (체크리스트 9종)
        │
   4. 포맷 변환 (citation 단위로 분해)
        │
   5. 검수 (자동 PII·citation·모순 + 수동 승인)
        │
   6. AI 커스터마이징 (어조·금기어·라우팅)
        │
   7. Sandbox 테스트 (100문항)
        │   └─ 임계 미달 → 4로 복귀
   8. Production 활성화 (Canary 10% → 50% → 100%)
        │
        ▼
[B. 운영 중 사이클 — 지속]
   ├─ 패치 동기화 (게임 업데이트 시)
   ├─ FAQ 보강 루프 (KB 미스 알림 → 작성)
   ├─ 만료 콘텐츠 정리 (시즌·이벤트 종료)
   └─ 정기 검수 (월 1회)
        │
        ▼
[C. 운영자 인사이트]
   ├─ 정확도 대시보드 (자기 tenant 한정)
   ├─ KB 미스 자동 알림
   └─ 응답 샘플 리뷰 (주 1회)
```

## 2. 초기 셋업 (A 단계)

### 2.1 Tenant 생성

| 항목 | 내용 |
|---|---|
| **액터** | `cross_admin` (요청은 게임사 책임자) |
| **입력** | 게임사 정보, 책임자 연락처, 계약·요금 플랜 |
| **출력** | `tenant_id`, 격리된 환경, 초기 `tenant_admin` 계정 |
| **게이트** | 계약 체결 + DPO 컴플라이언스 체크리스트 통과 |
| **소요** | 사람 작업 + 시스템 프로비저닝 |

체크리스트 (DPO):
- 개인정보처리방침에 CS Agent 명시
- `consent.cs_basic` 동의 흐름 합의
- `consent.cs_ai_training` 동의 별도 축 합의
- 데이터 보존·삭제 정책 합의 (PII_MASKING §6)
- 다국가 운영 시 거주 국가 규제 매트릭스 합의

### 2.2 메타데이터 등록

**액터**: `tenant_admin`

| 분류 | 항목 |
|---|---|
| **게임 기본** | 이름, 장르(RPG / 캐주얼 / MMO / 소셜 / 보드 등), 플랫폼(iOS / Android / PC / Web / Console) |
| **지역·언어** | 출시 국가 목록, 지원 언어 우선순위 |
| **결제** | 결제 시스템(Stripe / IAP / 자체), 환불 정책 요지, 가격 통화 |
| **시즌·운영** | 시즌 길이, 라이브 운영 주기, 정기 업데이트 일정 |
| **신원** | Firebase Sub 사용 / Game UUID 사용 / 양쪽 |
| **외부 채널** | Discord 서버 ID (선택), 이메일 도메인 (선택) |
| **AI 톤** | 캐주얼 / 격식 / 판타지 RPG체 / SF / 코믹 등 — 라이브러리 선택 + 자유 prefix |
| **에스컬레이션 커스텀** | 기본 5단계 외 게임별 룰 (예: 길드 분쟁 → 모더 큐) |

**게이트**: 필수 항목 모두 채워질 때까지 다음 단계 진행 불가.

### 2.3 콘텐츠 수집 체크리스트

**액터**: `tenant_admin` (계획) + `tenant_operator` (작성)

#### P0 (출시 전 필수)
- **FAQ 풀** — 최소 100문항, 카테고리(C1-C5) 분포 균등 권장
- **운영·환불 정책** — 환불 가능 조건, 분쟁 처리 절차, ToS, 개인정보 동의
- **결제 상품 카탈로그** — 상품명·가격·내용물·획득 방법·만료
- **계정 정책** — 가입·탈퇴·계정 복구·연동(Google/Apple/SNS) 정책

#### P1 (권장)
- **기획서 요약본** — 게임 세계관·시스템·주요 흐름 (원본 통째로 X)
- **패치노트 최근 6개월** — 현재 상태 파악
- **캐릭터·아이템·스킬 사전** — NER용 + 답변 인용용
- **이벤트·시즌 캘린더** — 현재 + 향후 4주

#### P2 (선택)
- 알려진 버그 임시 정책
- 디바이스별 트러블슈팅 가이드
- 커뮤니티 가이드라인

**게이트**: P0 모두 입력 + 인덱싱 완료.

### 2.4 콘텐츠 포맷 변환

**액터**: `tenant_operator`

KB 단위 메타데이터 표준:

| 필드 | 설명 |
|---|---|
| `title` | 문서 제목 |
| `slug` | URL-safe 식별자 (인용용) |
| `category` | C1-C5 또는 cross-cutting |
| `language` | ISO 코드 (단일 또는 다중) |
| `version` | "v0.3" 등 |
| `valid_from` | 유효 시작 |
| `valid_until` | 만료 (선택, 시즌·이벤트용) |
| `source` | 원본 출처 (기획서 §, 패치노트 ID 등) |
| `tags` | 검색 보조 태그 |

**분해 단위**: "한 질문에 한 답" 또는 "한 정책에 한 섹션" — citation 가능한 크기.

**대용량 기획서 처리**:
- 원본 통째 인덱싱 ❌ (인용 어려움, PII 위험, 무관 내용 노이즈)
- 권장: 운영팀이 "유저가 물어볼 만한 단위"로 발췌·재작성 후 인덱싱

**게이트**: 모든 문서가 citation 가능한 단위인지 자동 검사 통과.

### 2.5 검수

#### 자동 검사
| 검사 | 기준 |
|---|---|
| **PII 스캐너** | Tier 1+2+3 통과 필수 (PII_MASKING §2) |
| **Citation 가능성** | 제목·버전·섹션 메타데이터 누락 없음 |
| **언어 일치** | 선언 언어와 본문 언어 일치 (감지) |
| **모순 검사** | CROSS 레이어와 충돌(예: tenant "48시간 환불" vs CROSS "24시간 환불") → 경고 |
| **중복 검사** | 동일 슬러그·유사 본문 중복 → 경고 |
| **길이 검사** | 한 문서 N자 초과 시 분해 권장 |

#### 수동 검수
- `tenant_admin`이 자기 game의 모든 P0 자료 검토·승인.
- 작성자 ≠ 승인자 (SoD).
- CROSS 레이어 모순 발견 시 `cross_admin`과 조율.

**게이트**: 자동 PASS + 수동 승인. 둘 다 통과해야 인덱싱.

### 2.6 AI 게임별 커스터마이징

**액터**: `tenant_admin` + `cross_admin`

| 항목 | 내용 |
|---|---|
| **어조 프로파일** | AI_POLICY §1.2 기반. 라이브러리에서 선택 + 자유 prefix 추가 가능 |
| **금기어·고유명사 사전** | 게임 내 NPC명이 시스템 단어(예: "관리자")와 충돌 시 표기 규칙 |
| **카테고리 분류기 힌트** | 게임 특수 카테고리 예: "PVP 매치 보상 분쟁", "길드 마스터 권한 이양" |
| **에스컬레이션 라우팅 커스텀** | 기본 5단계 외 룰 (예: 길드 분쟁은 모더 큐 직행) |
| **첨부 정책** | 스크린샷 필수 여부, 동영상 허용 여부, 파일 크기 한도 |
| **응답 길이** | AI_POLICY §2 기본(6문장)에서 tenant별 조정 가능 |

**게이트**: `tenant_admin` 승인 + `cross_admin` 검토 (커스텀 라우팅 변경 시 필수).

### 2.7 Sandbox 테스트

**액터**: `tenant_admin` + `tenant_operator`
**환경**: production과 분리된 sandbox tenant

#### 시드 문항 구성 (100개)
| 출처 | 수량 |
|---|---|
| CROSS 제공 표준 문항 (카테고리·다국어 cover) | 50 |
| Tenant 게임별 문항 (자기 게임 특수성) | 50 |

#### 검증 메트릭
| 메트릭 | 임계 |
|---|---|
| 자동 해결률 (FAQ 카테고리) | ≥ 50% |
| Citation 부착률 (사실 주장 응답) | 100% |
| PII 누설 | 0건 |
| 카테고리 분류 정확도 | ≥ 85% |
| 다국어 일치율 | ≥ 95% |
| 에스컬레이션 라우팅 정확도 | ≥ 95% |
| 안전 민감 키워드 high priority 트리거 | 100% |

#### 출력
- PASS / 보강 필요 항목 리포트
- KB 미스 발생 시 어떤 질문이 미스였는지 목록

**게이트**: 모든 메트릭 임계 통과. 미달 시 §2.4(콘텐츠 보강) 또는 §2.6(커스터마이징)으로 복귀.

### 2.8 Production 활성화

**액터**: `cross_admin` (2인 승인) + `tenant_admin` 동의

#### Canary 단계
| 단계 | 트래픽 | 기간 | 게이트 |
|---|---|---|---|
| Canary | 10% (또는 단일 빌드) | 24시간 | 자동 해결률·오답·PII 모두 정상 |
| Stage 2 | 50% | 72시간 | 동상 + 운영팀 샘플 리뷰 OK |
| Full | 100% | — | — |

#### 모니터링
- 시간당 자동 해결률, 오답 보고, 에스컬레이션 비율 추이
- 알람 임계: 자동 해결률 < 30%, 오답 > 5%, PII 누설 1건이라도

#### 롤백 트리거
- 자동 해결률 30% 미만 지속
- PII 누설 발견
- 시스템 장애 / LLM 응답 거부 비율 급등
- `cross_admin` 또는 `tenant_admin` 어느 쪽이든 중단 요청

## 3. 운영 중 사이클 (B 단계)

### 3.1 패치 동기화

| 항목 | 내용 |
|---|---|
| **트리거** | tenant가 게임 업데이트 배포 (이상적으로 24시간 전 KB 갱신) |
| **액터** | `tenant_operator` + `tenant_admin` 승인 |
| **작업** | 패치노트 업로드 / 신규 캐릭터·아이템·시스템 KB 추가 / 정책 변경 / 알려진 버그 임시 정책 |
| **프로세스** | §2.4 → §2.5 → 인덱싱 (경량화). Sandbox 미니 테스트 (10문항) 권장 |
| **반영 lag** | < 1시간 (PRD §7.2) |

### 3.2 FAQ 보강 루프

| 항목 | 내용 |
|---|---|
| **입력** | AI 오답 보고, KB 미스 패턴, 에스컬레이션 빈도 |
| **시스템 제안** | "이 질문 / 유사 질문이 지난 7일 N회 KB 미스 → KB 추가 권장" |
| **액터** | `tenant_operator` 작성 → `tenant_admin` 승인 |
| **SLA** | 권장 알람 후 7일 이내 처리 (작성 / "사람만 답함" 표시 / 무시) |

### 3.3 만료 콘텐츠 정리

- KB 문서의 `valid_until` 도달 → 자동 비활성 (검색 결과 제외).
- 시즌·이벤트 종료 트리거 → 관련 태그 일괄 비활성.
- 비활성된 문서를 인용한 응답은 발생 안 함.
- 비활성 6개월 후 자동 archive (영구 보존 옵션 별도).

### 3.4 정기 검수

| 주기 | 월 1회 |
|---|---|
| **액터** | `tenant_admin` + 보안팀 PII 샘플 |
| **점검 항목** | 마지막 검토 90일 초과 문서 / 응답 인용 0인 문서 / CROSS 레이어 모순 / 깨진 인용 / PII 누락 의심 |
| **출력** | 보강·삭제·통합 액션 리스트 |
| **미수행 시** | 자동 경고. 90일 이상 미검수 시 알람 (ONB-10 결정) |

## 4. 운영자 인사이트 (C 단계)

### 4.1 답변 정확도 대시보드

**대상**: `tenant_admin`, `tenant_operator` (자기 tenant 한정)

| 메트릭 | 시각화 |
|---|---|
| 자동 해결률 | 시계열 + 카테고리별 |
| 카테고리 분포 | 파이 / 시계열 |
| 평균 첫 응답 시간 | 채널별 |
| 에스컬레이션 비율 | 카테고리별 |
| 오답 보고 비율 | 시계열 |
| 다국어 분포·일치율 | 언어별 |
| KB 미스 빈도 | 질문 클러스터 리스트 |

### 4.2 KB 미스 자동 알림

- **룰**: 동일·유사 질문이 7일 내 5회 이상 KB 미스 (의미 임베딩 클러스터링)
- **알림**: `tenant_admin`에게 "이 질문에 대한 KB 추가 권장" + 후보 질문 예시 묶음
- **액션**: 무시 / 작성 / "이 질문은 사람만 답함" 표시 (영구 사람 라우팅)

### 4.3 AI 응답 샘플 리뷰

- **주기**: 주 1회
- **샘플**: 카테고리별 무작위 10건씩 (총 50건)
- **평가**: `tenant_operator`가 1-5점 + 오답 사유 태그 (KB 부족 / 톤 일탈 / Citation 누락 / 분류 오류 등)
- **활용**: 누적 데이터로 시스템 프롬프트·KB 개선 지표 산출

## 5. 데이터 거버넌스 (D 단계)

### 5.1 콘텐츠 소유권

- tenant가 작성한 KB는 tenant 소유.
- CROSS는 인프라(저장·인덱싱·검색·임베딩 계산)만 제공.
- tenant 탈퇴 시 데이터 export 제공. 동의 시 archive, 미동의 시 삭제.

### 5.2 자동 PII 차단

- 모든 KB 입력은 PII 스캐너 필수 통과 (PII_MASKING §3.4).
- 통과 못한 문서는 인덱스 등록 거부 + 작성자에게 사유 회신.
- 위반 패턴 누적되면 작성자 권한 일시 제한 (보안팀 검토 후 복원).

### 5.3 버전 관리

- 모든 KB 변경 이력 보존.
- 롤백 가능 (이전 버전으로 즉시 복원).
- 변경 작성자·승인자·시각 기록 (감사 로그, RBAC §6).

### 5.4 승인 워크플로 (SoD)

| 단계 | 액터 |
|---|---|
| 작성 | `tenant_operator` |
| 자동 검수 | 시스템 |
| 수동 승인 | `tenant_admin` (또는 `cross_admin`) |
| 긴급 변경 | `cross_admin`이 작성·승인 통합 가능 (사후 보고 필수) |

**동일인 작성·승인 금지**. 시스템이 검출 시 거부.

### 5.5 CROSS 레이어와의 관계

- CROSS 레이어(지갑·결제·온체인 매뉴얼)는 `cross_admin` 소유·관리.
- tenant는 CROSS 레이어 검색만, 편집 불가.
- 모순 발생 시 우선순위 룰: **Q-011-B**에 의거, tenant 우선 또는 CROSS 우선 별도 결정 (현재 미정).
- CROSS 레이어 변경 시 모든 tenant에 영향 → 변경 전 사전 공지 + shadow mode 권장.

## 6. 학습 자료 카탈로그 (참고용)

게임사가 사전 준비해두면 좋을 자료 형태 정리.

| 자료 | 권장 형식 | KB 분해 단위 | 비고 |
|---|---|---|---|
| FAQ 풀 | 질문 + 답안 + 카테고리 + 태그 | 1 QA = 1 문서 | 최소 100건 |
| 운영 정책 | 마크다운 / 위키 | 1 정책 섹션 = 1 문서 | 환불·계정·결제 분리 |
| 패치노트 | 버전별 변경사항 | 1 버전 = 1 문서 또는 1 변경 = 1 문서 | 최근 6개월 |
| 캐릭터·아이템 사전 | 표 형태 | 1 항목 = 1 문서 (또는 묶음) | NER 사전 + 인용 |
| 결제 상품 | 카탈로그 표 | 1 상품 = 1 문서 | 만료일 메타데이터 필수 |
| 이벤트 캘린더 | 일정 표 | 1 이벤트 = 1 문서 | `valid_until` 필수 |
| 기획서 | 요약본 (원본 X) | 시스템·메커니즘 단위 발췌 | 유저 질문 가능한 단위 |
| 알려진 버그 | 임시 정책 | 1 버그 = 1 문서 | 임시 표시·해결 시 archive |
| 디바이스 트러블슈팅 | OS·기기별 | 1 OS·1 증상 = 1 문서 | 선택 |
