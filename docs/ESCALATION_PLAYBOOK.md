## 🚨 출시 시점 운영 모델 (M1, 2026-06)

**휴먼 상담사 0명**. 본 플레이북의 큐 분류·우선순위·SLA 정의는 **향후 인력 확보 시 적용**되는 기준이다. 출시 시점에는:

- **모든 큐를 `cross_admin` 단독 처리** (가용 시간에 한해).
- 사람 처리 SLA **미보장**. AI 응답 SLO만 보장 (p95 < 10초).
- 유저에게 "추후 처리" 안내, 구체 시간 미명시.
- 안전 민감 (자해·법적·위협) high priority는 **cross_admin 즉시 알람** (PagerDuty/Slack).
- 그 외 큐는 **일 1회 다이제스트**로 cross_admin에게 상황만 통지.
- `cross_payment` / `cross_infra` / `cross_moderation` / `cross_general` 역할은 정의되어 있으나 **인력 0명**.

본 문서의 §3 큐별 플레이북 / §4 SLA / §5 핸드오프 등은 **향후 인력 확보 시 가이드**로 그대로 유지.

## 0. 핵심 원칙

1. **AI 초안은 참고용** — 그대로 보내도 되지만, 책임은 운영자.
2. **회신 전 사실 확인** — AI 툴 결과를 운영자가 한 번 더 검증 (특히 결제·온체인).
3. **변경 액션은 별도 시스템** — CS Agent는 안내·확인만. 실제 환불·재지급·계정 변경은 운영자가 자기 도구로 수행.
4. **유저 언어 유지** — 운영자가 직접 회신할 때도 유저 언어로 (번역 도구 사용 OK).
5. **종결 사유 명시** — 모든 close 액션은 사유 태그 필수 (분석용).
6. **24시간 SLA** — 첫 응답 24시간 미준수 시 자동 재할당 + priority 상승.

## 1. 큐 구조 개요

```
[유저 → AI Agent]
      │
      ▼
[에스컬레이션 발생 — TRIAGE-004 우선순위]
      │
      ├─① "사람과 대화" 즉시 사람
      ├─② 결제·환불 키워드        → CROSS 결제 큐
      ├─③ 지갑·복구 키워드        → CROSS INFRA 큐
      ├─④ 어뷰징·신고            → CROSS 모더 큐
      ├─⑤ 버그 리포트 + 기타      → Tenant 운영자 큐
      └─⑥ tenant 부재·미응답     → CROSS 일반 fallback 큐
```

5개 큐:

| 큐 | 담당 역할 | 주 카테고리 |
|---|---|---|
| **Tenant 운영자 큐** | `tenant_operator` | 게임 진행 FAQ 미스, 버그, 일반 |
| **CROSS 결제 큐** | `cross_payment` | 환불·결제 실패·미지급·결제 분쟁 |
| **CROSS INFRA 큐** | `cross_infra` | 지갑·온체인·tx 실패·시드 분실 |
| **CROSS 모더 큐** | `cross_moderation` | 어뷰징·신고·법적 위협·Sentinel appeal |
| **CROSS 일반 fallback 큐** | `cross_general` | tenant 부재·미응답·미분류 |

## 2. 공통 — AI Handoff Package

운영자가 티켓을 열면 다음이 함께 보여진다.

### 2.1 핸드오프 패키지 구성
| 항목 | 설명 |
|---|---|
| **대화 전체** | 유저↔AI 메시지 시계열 |
| **유저 프로파일 미니뷰** | Profile-primary 컨텍스트: 신원 + 최근 결제 + 최근 온체인 + 진행도 + 밴 여부 |
| **AI 분류 결과** | 카테고리·confidence·언어 |
| **툴 호출 로그** | 어떤 툴을 어떤 인자로 불렀고 어떤 응답을 받았는지 |
| **AI 초안 (있는 경우)** | 발송 전 보류된 응답안 |
| **에스컬레이션 사유** | `reason`, `priority`, `summary`, `suggested_action` |
| **첨부** | 스크린샷·로그·동영상 URL |
| **유사 과거 티켓** | 같은 유저의 과거 6개월 티켓 (있을 때) |

### 2.2 운영자 1차 액션 (모든 큐 공통)
1. **5초 스캔** — 카테고리·priority·요약 빠르게 인지.
2. **사실 확인** — AI 툴 결과 vs 본인 도구 결과 대조 (특히 결제·온체인 금액·시각).
3. **claim** — 본인이 처리할 거면 즉시 claim (큐에서 제거, 다른 운영자 중복 처리 방지).
4. **추가 정보 필요한가** — yes → 유저에게 질문 / no → 처리 단계.

## 3. 큐별 플레이북

### 3.1 Tenant 운영자 큐

#### 주 시나리오
- KB 미스 FAQ
- 클라이언트 크래시 / 버그
- 진행 막힘 (시스템 이해 부족)
- 게임 특수 정책 문의

#### 처리 단계
1. **AI 초안 검토** — 그대로 보낼 만한지 / 보강 필요한지.
2. **버그면** — 재현 단계 정리, 빌드·디바이스·시각 확인, 개발팀 이슈트래커에 등록 (Linear/Jira/Slack — OQ-7).
3. **FAQ 미스면** — 정답 회신 + KB 미스 알림 큐로 등록 (TENANT_ONBOARDING §3.2 보강 루프).
4. **회신 → resolve** — 종결 사유 태그 (resolved_by_human / forwarded_to_dev / kb_gap).

#### 회신 예시
- 버그 접수: "확인했습니다. 빌드 X.Y에서 재현 가능한 이슈로 보입니다. 개발팀에 정식 접수했고, 후속 안내는 이메일로 드리겠습니다."
- KB 미스 FAQ: "답변드릴게요. {정답}. (관련 가이드는 다음 패치 노트에 추가 예정입니다)"

#### 종결 사유 태그
- `resolved` — 운영자가 직접 해결
- `forwarded_to_dev` — 개발팀 이관 완료
- `kb_gap_filled` — 답변 후 KB 추가 예정
- `duplicate` — 동일 이슈 진행 중 티켓 있음
- `cannot_reproduce` — 재현 불가, 유저 확인 요청

### 3.2 CROSS 결제 큐

#### 주 시나리오
- 환불 요청 (정책 내·정책 외)
- 결제 실패·중복 결제
- 결제 후 아이템 미지급
- 결제 수단 변경 / 카드 정보 분쟁

#### 처리 단계
1. **AI 핸드오프 + 결제 시스템 양쪽 대조** — 거래 ID·금액·시각·상태.
2. **정책 매칭** — 환불 가능 조건(tenant 정책 + CROSS 표준) 적용.
3. **결정 트리**:
   - 정책 내 환불 → 결제 시스템에서 환불 실행 (별도 도구) → 유저 회신
   - 정책 외 환불 요청 → 거절 회신 + 사유 명시 (정중) / 또는 예외 승인 요청 (2인 승인)
   - 미지급 → 게임 백엔드에 재지급 요청 (별도 도구) → 유저 회신
4. **회신 → resolve**.

#### 회신 예시
- 환불 승인: "확인했습니다. 5/10 결제 49.99 USD 환불 처리되었습니다. 영업일 기준 3~5일 내 카드사 반영 예정입니다."
- 환불 거절: "안타깝지만 본 결제는 환불 정책({조건})에 해당하지 않아 환불이 어렵습니다. 자세한 사유: {설명}"
- 미지급 → 재지급: "결제 확인 후 누락된 {상품} 지급 완료했습니다. 게임 재접속 후 확인 부탁드립니다."

#### 종결 사유 태그
- `refund_approved`, `refund_denied`, `re_issued`, `policy_exception` (2인 승인 필수), `fraud_suspected` (보안팀 통보)

#### 주의
- 환불 약속은 실제 환불 처리 후에만.
- 챗 기록상 AI가 환불 약속한 흔적 있으면 책임 명확히 (AI는 약속 못 함 — AI_POLICY §1.4).
- 사기·중복 결제는 보안팀 통보 (사기 패턴 학습).

### 3.3 CROSS INFRA 큐

#### 주 시나리오
- 트랜잭션 미동기화 / 실패
- 민팅 실패 / 가스비 부족
- 지갑 분실·복구 요청
- 시드 노출 사고

#### 처리 단계
1. **AI 핸드오프 + 온체인 익스플로러 대조** — tx hash 직접 조회.
2. **결정 트리**:
   - tx 정상 confirmed → "곧 동기화될 것" 안내 (이미 AI가 답했을 가능성)
   - tx pending → 네트워크 상태 안내, 재시도 가이드
   - tx failed → 사유 진단 (가스 부족·revert·nonce 등), 재시도 안내
   - 지갑 분실 → KYC·본인확인 절차 안내 (별도 워크플로)
   - **시드 노출** → priority high, 즉시 자산 동결·이체 안내, 보안팀 통보
3. **회신 → resolve**.

#### 회신 예시
- tx confirmed: "tx 0xAb12 정상 confirmed 상태입니다. 인게임 동기화는 보통 5분 이내. 추가 대기 후에도 미반영이면 다시 알려주세요."
- 시드 노출: "지금 즉시 다음 단계를 따라주세요: ① 모든 자산을 새 지갑으로 이체 ② 기존 지갑·계정 비활성화 신청 ③ 변경된 지갑 주소로 연동 갱신. 보안팀이 직접 도와드리겠습니다."

#### 종결 사유 태그
- `tx_normal`, `tx_failed_resolved`, `wallet_recovery_initiated`, `seed_exposed_handled`, `requires_kyc`

#### 주의
- 온체인은 비가역. 추측 답변 절대 금지.
- 가스비 환급·재시도는 정책에 따라 처리 (예외 시 2인 승인).
- 자산 가치 임계 (예: $1000+) 사고는 자동 high priority.

### 3.4 CROSS 모더 큐

#### 주 시나리오
- 핵·매크로 신고
- 욕설·괴롭힘·위협
- 사기 거래 신고
- 부정 결제 신고
- Sentinel appeal (밴 항소)

#### 처리 단계
1. **신고자 안전 우선** — 신고자의 다른 정보는 가림 (RBAC §8.1).
2. **증거 검증** — 첨부·로그·동영상 검토.
3. **피신고자 조회** — 별도 권한으로 (cross_admin 승인 필요할 수 있음, RBAC §8.2).
4. **판정**:
   - 위반 확인 → 제재 워크플로 (별도 도구) → 신고자 회신 (결과 비공개 원칙)
   - 증거 부족 → 추가 정보 요청 또는 보류
   - 위반 아님 → 종결
5. **Sentinel appeal** → 원래 밴 사유 검토 후 유지·해제 결정.

#### 회신 예시 (신고자)
- "신고 접수 후 모더팀이 비공개로 검토했습니다. 적절한 조치가 이루어졌으며, 구체적인 결과는 정책상 공유드리지 않는 점 양해 부탁드립니다."
- 추가 정보 요청: "조사를 위해 추가 정보가 필요합니다: 발생 시각, 채널/장소, 가능하면 추가 스크린샷."

#### 회신 예시 (피신고자 — appeal)
- 해제: "검토 결과 제재 사유에 오류가 확인되어 해제 처리했습니다. 불편을 드려 죄송합니다."
- 유지: "재검토 결과 제재 사유가 유효하다고 판단했습니다. 자세한 사유: {요지}"

#### 종결 사유 태그
- `violation_confirmed_action_taken`, `insufficient_evidence`, `no_violation`, `appeal_upheld`, `appeal_overturned`, `referred_to_legal`

#### 주의
- 피신고자 정보를 신고자에게 노출 절대 금지.
- 법적 위협·자해 키워드 즉시 법무팀·보안팀 협의.
- 판정 근거는 감사 로그에 명확히 (사후 분쟁 대비).
- 신고자 개인정보가 피신고자에게 노출 안 되도록 회신 본문 검수.

### 3.5 CROSS 일반 fallback 큐

#### 주 시나리오
- tenant 운영자 24시간 미응답 → 재할당
- 카테고리 명확치 않음
- 신규 tenant 운영 미숙

#### 처리 단계
1. **재라우팅 판단** — 정확한 큐 식별되면 즉시 핸드오프 (3.1~3.4).
2. **즉시 응답 가능** — 일반적인 안내 가능하면 직접 회신.
3. **tenant 운영자 보강** — 자주 발생하면 tenant 측에 패턴 공유 (운영 컨설팅).
4. **회신 → resolve / forward**.

#### 종결 사유 태그
- `resolved`, `forwarded_to_correct_queue`, `tenant_coached`, `unresolved_systematic` (시스템 이슈로 분석)

## 4. SLA · 재할당 · 우선순위

### 4.1 SLA
| 우선순위 | 첫 응답 SLA | 종결 SLA (목표) |
|---|---|---|
| high (안전 민감·자산 사고) | 1시간 | 24시간 |
| normal | 24시간 | 7일 |
| low | 48시간 | 14일 |

### 4.2 SLA 위반 시
- 24시간 무응답 → 자동 재할당 (큐 내 다른 운영자 또는 CROSS 일반 큐)
- 추가 24시간 무응답 → priority 1단계 상승 + Slack 알람 (tenant_admin / cross_admin)
- 48시간 무응답 (high) → 보안팀 자동 통보

### 4.3 우선순위 재조정
- 운영자가 처리 중 사안 중대성 발견 시 priority 상향 (사유 기록).
- AI 분류 오류 시 운영자가 카테고리 재분류 (감사 로그).

## 5. 큐간 핸드오프

### 5.1 핸드오프 트리거
- 잘못된 큐로 들어옴 (예: 환불처럼 보였는데 실제 게임 진행 문제)
- 처리 중 다른 큐 권한 필요 (예: 결제 큐에서 처리 중 어뷰징 정황 발견)
- 복합 이슈 (결제 + 어뷰징 동시) → 둘 다 협업

### 5.2 핸드오프 액션
1. 운영자가 "큐 변경" 액션 → 새 큐 선택 + 사유 입력.
2. RBAC §5에 따라 새 큐 담당 역할에 권한 부여, 원 담당은 읽기 잔존.
3. 새 큐 담당자에게 알림.
4. 유저에게는 핸드오프 사실 알리지 않음 (혼란 방지) — 단 SLA 카운터는 리셋되지 않음.

### 5.3 협업 (multi-queue)
- 한 티켓에 여러 큐 담당이 동시 attach 가능.
- 모더팀 + 결제팀 협업 등.
- 종결 권한은 1차 담당자 (또는 cross_admin).

## 6. 자주 발생하는 패턴별 가이드 (Cookbook)

### 6.1 "환불 안 됐어요" (실제로는 결제 자체 실패)
- 결제 시스템 조회 → 거래가 시작도 안 됨. 환불 대상 없음.
- 회신: "확인 결과 5/10 시도하신 결제가 실제로 진행되지 않았습니다. 카드사 측에서 일시 보류 후 자동 취소된 케이스로 보입니다. 추가 청구 발생 시 다시 알려주세요."

### 6.2 "민팅이 안 됐어요" (실제로는 가스 부족으로 revert)
- 익스플로러 확인 → revert 이력.
- 회신: "tx 확인했습니다. 가스 부족으로 revert된 상태로, 결제도 발생하지 않았습니다. 가스를 충분히 설정 후 재시도해주세요. 자세한 가이드: {링크}"

### 6.3 "PlayerX가 매크로" (실제로는 합법적 자동 플레이 기능)
- 게임 시스템에 정식 매크로 기능이 있으면 (예: 자동 사냥) 위반 아님.
- 회신: "신고 접수 후 검토했습니다. 본 게임은 자동 사냥 기능을 정식 지원하고 있어 PlayerX의 행동은 정책 위반에 해당하지 않습니다. 별도 조치는 없으나, 다른 우려 사항이 있으시면 알려주세요."

### 6.4 "내 계정 해킹당했어요"
- 우선순위 즉시 high.
- 처리: 비밀번호 초기화 안내 + 최근 로그 검토 + 최근 이체·결제 분석 + 자산 보호 조치.
- 보안팀 통보 (사기 패턴 학습).

### 6.5 "환불 안 해주면 고소할게요"
- 법적 위협 키워드. AI는 이미 사람 라우팅 했을 것.
- 처리: 정중·중립 톤. 정책 설명. 법무팀 통보 (대량 발생 시 패턴).
- 회신: "불편을 드려 죄송합니다. 본 건은 법무팀에 정식 이관하여 검토 후 추가 안내드리겠습니다."

### 6.6 동일 유저 반복 동일 문의
- AI 단계에서 이미 한 번 답변 → 유저가 같은 질문 반복.
- 사람 라우팅 → 운영자가 한 번 더 명확히 + 다른 채널 안내 + (반복적이면) tone-down.

### 6.7 "다른 유저 정보 알려주세요"
- AI는 거부 (AI_POLICY §3.5). 사람 라우팅됐을 수도.
- 운영자도 거부. "본인 외 정보는 안내드릴 수 없습니다."

## 7. 종결·재오픈

### 7.1 종결 (resolved → closed)
- `resolved` 상태로 일정 시간 (예: 7일) 후 자동 `closed`.
- 유저가 그 안에 추가 메시지 보내면 재오픈.

### 7.2 재오픈
- 종결 후 30일 이내 → 같은 티켓으로 재오픈.
- 30일 초과 → 새 티켓 생성, 단 과거 티켓 참조.

### 7.3 종결 사유 통계
- 종결 사유 태그는 운영 인사이트의 핵심 데이터.
- 월간 리뷰 시 패턴 분석 (어떤 카테고리에서 어떤 종결 사유가 많은지 → KB·정책 개선).
