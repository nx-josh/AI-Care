## 1. 배경 및 문제 정의

### 1.1 현황
- 게임 서비스를 운영하면서 발생하는 CS 문의가 다양한 채널(Discord, 인게임, 이메일)을 통해 유입.
- 1차 분류·간단 응답에 다수의 휴먼 상담사 리소스 투입.
- 24시간 글로벌 서비스 특성상 야간·다국어 응답 커버리지가 부족.
- 단순 FAQ 응답까지 상담사 큐에 쌓여 복잡한 사례의 응답이 지연됨.

### 1.2 해결하고자 하는 문제
1. **응답 지연** — 야간/주말/언어 미스매치로 인한 대기시간.
2. **단순 문의 비중** — FAQ·중복 문의가 상담사 자원을 소모.
3. **다국어 응답 한계** — 글로벌 유저 대응 인력 부족.
4. **데이터 단편화** — 채널별로 분리된 문의 이력, 통합 가시성 부재.

### 1.3 솔루션 방향
AI Agent가 **1차 분류·응답·정보 수집**을 자동화하고, **변경이 필요한 모든 액션은 휴먼 상담사에게 안전하게 에스컬레이션**한다. 모든 채널을 통합한 단일 큐와 대시보드로 운영 가시성 확보.

## 2. 목표 및 비목표

### 2.1 목표 (Goals)
- **G1.** CS 문의의 1차 응대를 자동화하여 휴먼 상담사 부담 감소 (정량 목표는 §6 메트릭).
- **G2.** 글로벌 유저 대상 다국어 24/7 1차 응답 제공.
- **G3.** 변경 액션은 100% 휴먼 검토를 거치는 **안전한 에스컬레이션** 보장.
- **G4.** 3개 채널(Discord, 인게임 SDK, 이메일)을 단일 코어에서 통합 처리.
- **G5.** 모든 대화·툴호출·에스컬레이션을 영속 저장하여 감사·분석 가능.

### 2.2 비목표 (Non-goals, v1 제외)
- AI가 환불·계정 제재·아이템 지급·비밀번호 재설정 등 **mutating 액션을 직접 실행하지 않는다**.
- 음성·전화 상담 채널 미지원 (텍스트 전용).
- 카카오톡 채널은 v2로 이연 (카카오 비즈 채널 심사·계약이 별도 트랙).
- AI가 게임 플레이 가이드를 **창작**하지 않는다. KB에 있는 검증된 정보만 인용.

| **CS 상담사** | AI 응답 검토, 변경 액션 실행, 큐 처리 | 웹 대시보드 | ❓ 결정 필요 (한국어 단일? 다국어?) |

- 플레이어는 Firebase 또는 게임 클라이언트를 통해 인증된 상태로 채널에 접근 가능.

## 6. 성공 지표 (Success Metrics)

### 6.1 정량 지표 (❓ 목표 수치는 운영팀과 합의 필요)

| 지표 | 측정 방법 | v1 목표 (제안) |
|---|---|---|
| **자동 해결률** (Deflection rate) | 에스컬레이션 없이 종결된 티켓 비율 | ❓ 40% |
| **1차 응답 시간 P50** | 유저 메시지 → 첫 AI 응답까지 | < 5초 |
| **1차 응답 시간 P95** | 동상 | < 10초 |
| **상담사 큐 처리 시간 감소** | 도입 전후 큐 평균 대기시간 비교 | ❓ -50% |
| **다국어 커버리지** | 입력 언어 = 응답 언어 비율 | ≥ 95% |
| **잘못된 변경 액션** | AI가 변경 액션 실행한 건수 | **0건 (절대 목표)** |
| **유저 만족도 (CSAT)** | 대화 종료 후 1-5 평점 | ❓ 평균 ≥ 4.0 |

### 6.2 정성 지표
- 상담사가 AI 초안을 수정 없이 그대로 보낼 수 있는 비율
- 잘못된 정보로 인한 사후 정정 케이스 수

## 8. 비기능 요구사항 (Non-functional)

| 카테고리 | 요구사항 |
|---|---|
| **성능** | 1차 응답 P50 < 5초, P95 < 10초 |
| **가용성** | 99.5%+ (월 다운타임 ~3.6시간 이하) |
| **확장성** | ❓ 동시 활성 티켓 수 / 일일 메시지 수 예상치 필요 |
| **보안** | 모든 API 인증 강제. 게임 서버 호출은 read-only, write 권한 일체 없음 |
| **안전** | mutating 액션은 0건. 위반 발생 시 알람 |
| **PII** | 로그·LLM 입력에 결제 정보·이메일 등 민감 데이터 마스킹 정책 ❓ 결정 필요 |
| **감사** | 모든 AI 결정 (분류, 툴 호출, 응답) 추적 가능 |
| **다국어** | franc 또는 동급 라이브러리로 자동 감지, 응답은 동일 언어 |
| **비용** | ❓ 월 LLM 토큰 비용 상한 / 메시지당 평균 비용 목표 필요 |

## 9. 식별·인증 모델

### 9.1 유저 식별 키
다음 두 가지를 단일 어댑터로 수용:
- **Firebase Sub** — Firebase Auth ID 토큰의 `sub` claim. 서버에서 Firebase Admin SDK로 검증.
- **Game UUID** — 게임 개발사가 발급한 UUID. 인게임 SDK 호출 시 HMAC 서명으로 위변조 방지.

### 9.2 채널별 신원 전달

| 채널 | 신원 전달 방식 |
|---|---|
| Discord | Discord User ID → 게임 계정 링크 매핑 필요 (❓ 매핑 테이블 어디서 관리?) |
| 인게임 SDK | Firebase ID 토큰 (Bearer) 또는 Game UUID + HMAC |
| 이메일 | From 주소 매칭 (저신뢰), 또는 본문 내 계정 ID 자기 신고 |

### 9.3 익명 케이스
- 미인증 유저(가입 전 결제 문의 등)는 anonymous 식별자로 티켓 생성. 상담사 검토 시 본인 확인 필요.

## 11. 제약사항 및 가정

### 11.1 제약사항
- AI는 게임 서버에 **쓰기 권한 없음** (보안 정책).
- 모든 상담사 액션은 감사 로그 필수.
- 다국어 응답이라도 KB 콘텐츠는 ❓ 영어 기준 + 번역 vs 언어별 KB 운영 (결정 필요).

### 11.2 가정
- 게임사가 read-only 사내 API 또는 DB 액세스를 제공.
- 상담사 인원은 ❓ N명 (시프트·운영시간 정의 필요).
- 일일 평균 문의량 ❓ X건 (용량 산정용).
