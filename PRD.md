## 0. 관계 명세 — CROSS Triage 확장

본 PRD는 **CROSS Triage 모듈 스펙의 확장 PRD**다. CROSS Triage의 핵심 원칙을 채택하되, 채널 스코프를 의도적으로 확장한다. CROSS Triage 본체의 단계론(M0/M1/M2)은 본 프로젝트에 적용하지 않으며, 단일 출시 스코프로 운영한다.

### 0.1 채택 (CROSS Triage 원칙)
- **Profile-primary 답변 모델** — 질문 + CORE-001 통합 프로파일(신원·진행도·결제·온체인) → 답변. KB-primary 아님.
- **Ticket·콘솔 기반 위에 AI** — AI는 가속 레이어. ticket 엔티티·콘솔 없이는 AI 통합 위치가 모호함.
- **읽기 전용 원칙** — AI는 조회·안내·확인만. 환불·재지급·계정 변경은 사람. 자동화 도입 시 cap·2인 승인·audit 강제.
- **2-layer RAG** — Tenant 레이어(자기 게임 FAQ) + CROSS 레이어(지갑·결제·온체인 매뉴얼). citation 강제, 출처 없으면 응답 거부.
- **Multi-tenant 격리** — 모든 데이터·라우팅·접근권한 tenant(game ID) 기준 격리.
- **차별축** — Web3/온체인 컨텍스트 + cross-game/cross-tenant 통합 프로파일. 일반 AI 1차 답변군은 commodity화 → 같은 축에서 경쟁 금지.
- **CS 대화 자체가 PII** — 보존 18개월(분쟁 7년), `consent.cs_ai_training` 별도 동의, AI 호출 전 마스킹.
- **에스컬레이션 5단계 우선순위** (TRIAGE-004) — §8.4에 그대로 채택.

### 0.2 확장 (CROSS Triage 스펙 deviation)
| TRIAGE 원문 | 본 PRD 입장 | 사유 |
|---|---|---|
| TRIAGE-001 non-goal: "전화·이메일 채널 (위젯만)" | **이메일·Discord 채널 포함** | 야간 시간대 대응, 글로벌 다국어 유저, Discord 네이티브 사용자 커버 |
| TRIAGE-004 non-goal: "전화·이메일 채널 라우팅" | **이메일·Discord 라우팅 포함** | 동상 |
| TRIAGE M0/M1/M2 단계론 | **단일 출시 스코프** | 본 프로젝트는 페이즈 분할 없이 한 번에 출시 |

채널 확장은 **각 채널별로 신원·SLO·PII·citation·multi-tenant 5개 우려를 명시적으로 mitigation**해야 하며 §5에서 채널별 정책을 박는다.

### 0.3 비채택
- 음성/전화 채널 — CROSS Triage T7과 동일하게 보류.
- AI Browser Agent — CROSS Triage T8과 동일하게 자체 백엔드 보유로 불필요.

## 1. 배경 및 문제 정의

### 1.1 현황
- 게임 운영팀이 다양한 채널(인게임 / Discord 커뮤니티 / 이메일)로 유입되는 CS 문의를 1차 분류·간단 응답에 다수의 휴먼 상담사 리소스 투입.
- 24시간 글로벌 서비스 특성상 야간·다국어 응답 커버리지 부족.
- 단순 FAQ 응답까지 상담사 큐에 쌓여 복잡한 사례 응답이 지연됨.
- 채널별로 분리된 문의 이력 → 통합 가시성 부재.
- Web3/온체인 컴포넌트(지갑·결제 분쟁·NFT 민팅 이슈)는 기존 SaaS CS 솔루션이 다루지 못함.

### 1.2 해결하고자 하는 문제
1. **응답 지연** — 야간/주말/언어 미스매치로 인한 대기시간.
2. **단순 문의 비중** — FAQ·중복 문의가 상담사 자원을 소모.
3. **다국어 응답 한계** — 글로벌 유저 대응 인력 부족.
4. **데이터 단편화** — 채널별로 분리된 문의 이력.
5. **온체인 분쟁** — 지갑·트랜잭션·민팅 관련 CS는 기존 SaaS로 처리 불가능.

### 1.3 솔루션 방향
AI Agent가 **CORE-001 통합 프로파일을 컨텍스트로 1차 응답·정보 수집**을 자동화하고, **변경이 필요한 모든 액션은 휴먼 상담사에게 안전하게 에스컬레이션**한다. 모든 채널을 단일 코어·단일 큐로 통합.

## 2. 목표 및 비목표

### 2.1 목표 (Goals)
- **G1.** CS 문의의 1차 응대를 자동화하여 휴먼 상담사 부담 감소.
- **G2.** 글로벌 유저 대상 다국어 24/7 1차 응답 제공.
- **G3.** 변경 액션은 100% 휴먼 검토. 향후 자동화 시에도 cap·2인 승인·audit 강제.
- **G4.** 3채널(SDK 위젯/Discord/이메일)을 단일 코어에서 통합 처리.
- **G5.** 모든 대화·툴호출·에스컬레이션을 영속 저장, tenant 격리.
- **G6.** Profile-primary 답변 모델로 KB-primary 경쟁사 대비 컨텍스트 우위 확보.

### 2.2 비목표 (Non-goals — v1)
- AI가 mutating 액션(환불·계정 변경·아이템 지급 등)을 직접 실행. 자동화 도입은 별도 명세 시 결정.
- 음성·전화 상담 채널 (영구 비채택).
- WEB 본체·Mobile 네이티브 앱·카카오톡 (v1 이후 별도 트랙 — §5.3 참조).
- AI 학습용 자체 모델 학습 — RAG로 제한.
- AI ↔ AI 협업.

## 3. 사용자 및 페르소나

| 페르소나 | 목적 | 주 채널 | 출시 시점 인력 |
|---|---|---|---|
| **플레이어 (유저)** | 본인 문제 해결, 신고, FAQ | SDK 위젯 / Discord / 이메일 | — |
| **`cross_admin`** | 시스템 전반 관리·전 큐 처리 (단독) | Hub Console | **출시 시 유일한 운영 인력** |
| **Tenant 운영자** | 자기 game ID 큐 처리 (향후 인력 확보 시) | Hub Console | **출시 시 0명** |
| **CROSS 결제·INFRA·모더팀** | 에스컬레이션 큐 처리 (향후) | Hub Console | **출시 시 0명** |
| **개발팀 엔지니어** | 버그 리포트 수신 (별도 채널) | 외부 이슈트래커 | — |

### 3.1 운영 모델 — 상담사 없음 (DECISIONS §1)
- **출시 시점 휴먼 상담사 0명**. AI 1차 응답이 사실상 전부.
- 에스컬레이션 큐는 적재되며 **cross_admin이 단독 처리**. 처리 시간 미확약.
- 사람 응답이 즉시 어려운 케이스는 "추후 처리" 안내. SLA 미보장.
- 향후 인력 확보 시 RBAC 매트릭스에 정의된 큐별 역할로 분배.

### 3.2 주요 가정
- 플레이어는 CORE-001 Profile Store에 등록된 유저 (Firebase Sub or Game UUID 보유).
- 미인증 유저는 anonymous 식별자로 티켓 생성. 본인 확인 전까지 변경 액션 금지.
- 모든 역할 정의(`tenant_operator`, `cross_payment` 등)는 향후 인력 확보 시 사용. M1 출시 시 cross_admin이 모든 역할 겸임.

## 5. 채널

### 5.1 v1 스코프

| 채널 | 신원 | 입력 방식 | 응답 방식 | 우선순위 |
|---|---|---|---|---|
| **SDK 위젯 (인게임)** | Firebase Sub (Bearer) 또는 Game UUID + HMAC | REST `POST /agent/chat` | JSON (스트리밍 옵션 OQ-12) | P0 — CROSS Triage 표준 |
| **Discord 봇** | Discord User ID → Profile 매핑 | Slash command / DM 스레드 | DM 스레드 답장 | P1 — 확장 |
| **이메일** | From 주소 매칭 + 본문 내 계정 ID 자기 신고 | 인바운드 웹훅 | 회신 메일 | P2 — 확장 (비실시간 정책 별도) |

### 5.1.1 비실시간 명시 (전 채널 공통)

**AI 응답은 즉시이나, 사람 처리는 모든 채널에서 비실시간**임을 유저에게 명확히 안내.

| 처리 단계 | 응답성 |
|---|---|
| AI 분류·응답 | 즉시 (p95 < 10초, 이메일은 30분) |
| 사람 처리 (환불·복구·신고 처리 등) | **비실시간** — SLA 미보장, "처리 대기 중" 안내 |

게임 내 SDK 위젯이라도 변경 작업(환불·계정 복구·아이템 재지급)은 즉답하지 않고 담당 큐로 적재.

### 5.2 채널별 5개 우려 mitigation

| 우려 | SDK (표준) | Discord (확장) | 이메일 (확장) |
|---|---|---|---|
| **신원 검증** | Firebase Admin SDK 토큰 검증 OR HMAC | OAuth 1회 매핑 후 Discord User ID → Profile (사전 링크 필요) | From 약함 → 본문에 계정 ID 자기 확인 단계 추가, 변경 액션 시 추가 인증 토큰 메일 발송 |
| **응답 SLO** | 채널 기본 ✓ | 채널 기본 ✓ | **별도 SLO** — 비실시간이므로 첫 응답 < 30분, 처리 < 24시간 |
| **Multi-tenant 격리** | 토큰의 tenant claim 또는 SDK가 `tenant_id` 헤더 전달 | Discord 서버 ↔ tenant 매핑 테이블 (`discord_guild_id → tenant_id`) | 발신 도메인 또는 받는 주소 prefix(`cs-{tenant}@`) ↔ tenant 매핑 |
| **PII 처리** | 위젯에 PII 입력 가이드, AI 호출 전 마스킹 | DM은 비공개이나 캐시 보존 ↑ → 자체 저장만, Discord 캐시 미신뢰 | **본문 평문 영수증·결제정보 가능성 ↑** → 자동 PII 스캐너 강제, 마스킹 후 LLM 호출 |
| **Citation 강제** | 위젯 UI에 출처 footer 렌더 | 임베드 메시지 footer로 출처 표시 | 메일 푸터에 출처 섹션. 출처 없으면 "사람 검토 중" 자동 응답으로 폴백 |

### 5.3 향후 확장 채널 (별도 트랙)
v1 출시 후 단계적으로 확장 예정. 본 PRD v1 스코프에는 포함되지 않으나, 데이터 모델·식별·RBAC 등은 향후 확장을 고려해 설계.

- **WEB 본체** — 자사 CS 포털(독립 surface). 게임 클라이언트 없이 유저가 직접 접근. cross-tenant 통합 가시성 가능.
- **Mobile 네이티브 앱** — iOS·Android 독립 앱. 푸시·오프라인 메시지 등 모바일 특화 기능.
- **카카오톡** — 채널 심사·계약 별도 트랙.

### 5.4 영구 비채택
- 음성·전화

## 6. 핵심 플로우 (Use Case Scenarios)

모든 플로우는 인게임 SDK 위젯 기준으로 작성. Discord·이메일은 동일 코어를 거치되 표현 채널만 다름 (§5.2 mitigation 적용).

### 6.1 결제 환불 문의 (C1)

**트리거**: 유저가 위젯에서 "5월 10일 49.99달러 결제 환불 부탁드려요"

| 단계 | 동작 |
|---|---|
| 1 | 위젯 → `POST /agent/chat` (Firebase ID 토큰 헤더) |
| 2 | 서버: 토큰 검증 → `tenant_id` + `profile_id` 추출 |
| 3 | 카테고리 분류기: `account` (결제 서브) |
| 4 | 통합 프로파일 inject: 신원 + `payments[]` (최근 10건) + `wallets[]` |
| 5 | AI: `payment_history` 툴 호출 → 5/10 결제 확인 (status: completed) |
| 6 | AI 답변: "5월 10일 결제 49.99 USD 확인했습니다. 환불은 결제팀 상담사가 처리합니다." |
| 7 | 자동 에스컬레이션: `reason=refund_request`, `priority=normal`, 라우팅 → **CROSS 결제팀** |
| 8 | 위젯 UI: AI 답변 + "결제팀에서 24시간 내 회신" 안내 + 출처 footer |

- **출처**: "CROSS Refund Policy v1.2 §3.1" (CROSS 레이어)
- **결과**: 정보 안내 + 자동 에스컬레이션 (refund 자동 실행 절대 금지)
- **예외**: 결제 ID 못 찾으면 "결제 내역 조회 실패 — 사람 상담 연결" 후 즉시 에스컬레이션

### 6.2 게임 진행 막힘 (C2 — FAQ)

**트리거**: "용병 5명 모았는데 다음 퀘스트 안 열려요"

| 단계 | 동작 |
|---|---|
| 1 | 위젯 → `/agent/chat` |
| 2 | 카테고리 분류기: `faq` |
| 3 | 통합 프로파일 inject: 진행도 (현재 퀘스트, 레벨, 캐릭터, 길드) |
| 4 | AI: `kb_search` 툴 (tenant 레이어 우선) — "용병 5명 다음 퀘스트" |
| 5a | KB 매치 → 인용하며 답변 |
| 5b | KB 미스 → "검증된 가이드 없음. 사람 상담 원하시면 버튼 눌러주세요" + escalate 옵션 |
| 6 | 위젯 UI: 답변 + 출처 footer |

- **출처 예**: "Pixel Heroes FAQ v0.3 §2.4 — Mercenary Quest Chain"
- **결과**: 자동 해결 (KB 매치 시), 에스컬레이션 없음
- **예외**: KB 미스 + 유저가 사람 요청 → tenant 운영자 큐
- **금지**: KB에 없는 정보를 추측·창작해서 답변하지 않음 (citation 없으면 응답 거부)

### 6.3 클라이언트 크래시 (C3 — 버그)

**트리거**: "던전3에서 자꾸 튕겨요" + 스크린샷 첨부

| 단계 | 동작 |
|---|---|
| 1 | 위젯 → `/agent/chat` (multipart, 스크린샷 포함) |
| 2 | 카테고리 분류기: `bug` |
| 3 | 통합 프로파일 inject: 신원 + 디바이스/세션/빌드 컨텍스트 (T1 자동 수집) |
| 4 | AI: `log_fetch` 툴 (since: 최근 30분) |
| 5 | AI: 재현 단계 정리 — "1) 던전3 입장 2) 첫 페이즈 3) 보스 등장 시 크래시 확인" |
| 6 | AI: 부족한 정보 수집 — "이전 빌드에서도 같은 현상이었나요?" |
| 7 | 정보 수집 완료 → 자동 에스컬레이션: `reason=bug_repro`, `priority=normal` |
| 8 | 라우팅: tenant 운영자 → 개발팀 이슈트래커 (OQ-7 결정) |
| 9 | 위젯 UI: "버그 접수 완료. 개발팀이 검토합니다. 진행 상황은 이메일로 안내" |

- **출처**: 없음 (조사용 인테이크, citation 비적용)
- **결과**: 항상 에스컬레이션 (FAQ 같은 자동 해결 없음)
- **예외**: 진행 불가 수준(계정 잠김, 결제 후 미지급) → `priority=high`

### 6.4 다른 유저 신고 (C4 — 어뷰징)

**트리거**: "PlayerX가 매크로 쓰는 거 같아요" + 동영상 URL 첨부

| 단계 | 동작 |
|---|---|
| 1 | 위젯 → `/agent/chat` |
| 2 | 카테고리 분류기: `abuse` |
| 3 | **AI는 신고 내용 판단 안 함** — 구조화 인테이크만 |
| 4 | AI: 누락 정보 수집 — 시간, 채널/장소, 추가 증거 |
| 5 | (피신고자 정보는 절대 노출 안 함 — Sentinel 데이터 보호) |
| 6 | 인테이크 완료 → 자동 에스컬레이션, priority 규칙: |
|  | • 위협·자해·욕설·딥페이크 → **high** |
|  | • 핵·매크로·사기 → normal |
| 7 | 라우팅: 모더레이션 큐 (권한 분리, Hub Builder RBAC) |
| 8 | 위젯 UI: "신고 접수됨. 모더팀이 비공개로 검토. 결과는 별도 공유되지 않습니다." |

- **출처**: 없음
- **결과**: 항상 에스컬레이션
- **예외**: 신고자가 본인 결제·계정 등으로 카테고리 잘못 진입 → 대화 중 재분류
- **연동**: SENTINEL-003 (Sentinel appeal ticket)도 동일 모더 큐로 통합

### 6.5 온체인 트랜잭션 미동기화 (C5 — 차별축)

**트리거**: "민팅 트랜잭션 5분째 안 들어와요" + tx hash `0xAb12...`

| 단계 | 동작 |
|---|---|
| 1 | 위젯 → `/agent/chat` |
| 2 | 카테고리 분류기: `onchain` |
| 3 | 통합 프로파일 inject: `wallets[]`, `onChain.mints[]`, `onChain.recentTx[]` |
| 4 | AI: `onchain_tx_status` 툴 → `0xAb12` 상태 확인 |
| 5a | tx confirmed → "`0xAb12` 정상 처리됨, 5분 안에 인게임 동기화 예정" |
| 5b | tx pending → "현재 pending. 네트워크 혼잡으로 X분 더 걸릴 수 있음" |
| 5c | tx failed → "트랜잭션 실패. INFRA 팀이 가스비·재시도 안내" + 에스컬레이션 |
| 6 | 위젯 UI: 답변 + tx 익스플로러 링크 + 출처 footer |

- **출처**: "CROSS On-chain Manual v1.0 §4.2 — Mint Sync"
- **결과**: 정상이면 자동 해결, 실패면 INFRA 팀 에스컬레이션
- **예외**: 지갑 자체 분실/복구 → 즉시 INFRA 팀 라우팅 (`priority=high` if 자산 가치 ≥ 임계)
- **차별축 표현**: 경쟁사 SaaS는 wallet/tx context 없음 → 본 시나리오 자체가 우위 영역

### 6.6 공통 규칙 (모든 플로우)

- **언어**: 입력 언어 자동 감지 → 동일 언어로 응답.
- **신원 없음**: anonymous 처리, 변경 액션 절대 금지, 본인 확인 단계 우선.
- **AI confidence < 임계**: 답변 생성하지 않고 사람 라우팅.
- **PII 마스킹**: LLM 호출 전 영수증·전화번호·이메일·지갑 시드 자동 마스킹.
- **출처 강제**: AI가 사실 주장 시 출처 미부착 → 응답 거부 (재시도 또는 에스컬레이션).
- **"사람과 대화" 버튼**: 위젯 어디서든 즉시 사람 라우팅 (라우팅 우선순위 ① §8.4).

## 10. 식별·인증 모델

### 10.1 유저 식별 키 (CORE-001 Profile Store)
- **Firebase Sub** — Firebase Auth ID 토큰의 `sub` claim. 서버에서 Firebase Admin SDK로 검증.
- **Game UUID** — 게임 개발사가 발급. 인게임 SDK 호출 시 HMAC 서명 강제.

### 10.2 채널별 신원 매핑
| 채널 | 매핑 방식 | mitigation |
|---|---|---|
| SDK | 토큰 직접 → Profile | 표준 경로 |
| Discord | `discord_user_id → profile_id` 매핑 테이블 (사전 1회 OAuth 링크) | 매핑 없으면 "계정 연결" 안내만 |
| 이메일 | From 주소 → Profile (다중 매칭 가능) | 변경 액션 시 추가 인증 메일 발송 |

### 10.3 Multi-tenant 신원
- 모든 식별 키는 **`(tenant_id, identity)` 쌍**으로 관리. 같은 Firebase Sub이라도 tenant 다르면 별 프로파일.

### 10.4 익명 케이스
- 미인증 유저는 `anonymous:{channel}:{session_id}`로 티켓 생성. 본인 확인 전까지 변경 액션 금지.
